# **PARTIE 1 — DÉLÉGATION & RESPONSABILISATION (3h30)**

---

# **1. La délégation dans les projets IA**

## **1.1. Pourquoi déléguer ?**

Dans une équipe IA, le chef de projet n’a pas un rôle d’expert sur tous les domaines (Data Engineering, Data Science, DevOps, ML, RGPD…).
Déléguer permet :

* d’éviter la surcharge mentale du chef de projet,
* de responsabiliser les profils techniques,
* de fluidifier la livraison (CI/CD, MLOps),
* d’encourager l’autonomie et l’innovation,
* de réduire les risques d’erreurs humaines (surtout dans les modèles sensibles).

La délégation n’est pas un transfert de tâche, mais **un transfert de responsabilité encadrée**.

## **1.2. Les obstacles courants à la délégation**

* “Ça ira plus vite si je le fais moi-même.”
* “Je ne suis pas sûr de leur niveau.”
* “C’est trop important pour être confié.”
* “Si je délègue, je perds du contrôle.”
* “Ils sont déjà surchargés.”

Ces croyances mènent à :

* surcharge du chef de projet,
* démotivation de l’équipe,
* faible progression des collaborateurs.

D’où l’importance d’un **cadre clair de délégation**.

---

# **2. Les 5 niveaux de délégation**

### **Niveau 1 – Exécution strictement guidée**

Le chef de projet dit : quoi, comment, quand.
Adapté aux juniors ou tâches critiques (RGPD, sécurité).

### **Niveau 2 – Exécution guidée + marge de manœuvre**

Le chef de projet explique pourquoi, propose une méthode mais laisse des choix.

### **Niveau 3 – Résultat attendu, méthode laissée libre**

Utilisé pour les profils expérimentés (Data Scientists, DevOps seniors).

### **Niveau 4 – Co-construction**

Le chef de projet discute du problème, le collaborateur propose une solution.

### **Niveau 5 – Délégation totale**

Le collaborateur prend la décision seul dans une zone clairement définie (ex : choix des hyperparamètres, pipeline de features).

---

# **3. Comment choisir le niveau de délégation ?**

On s'appuie sur :

1. **Compétence technique**
2. **Motivation**
3. **Confiance**
4. **Importance / sensibilité de la tâche**

Exemple IA :

* Développement d’un script d’évaluation → Data Scientist senior → niveau 5
* Définition du modèle de données → Data Engineer confirmé → niveau 3 ou 4
* Mise en place de la CI/CD IA → DevOps senior → niveau 4 ou 5
* Préparation dataset sensible (RGPD) → Data Engineer junior → niveau 1 ou 2

---

# **4. Encadrer la délégation : les “6C”**

1. **Clarté** (objectif, métriques, périmètre)
2. **Confiance** (exprimée explicitement)
3. **Compétence** (vérifier la maîtrise nécessaire)
4. **Cadre** (temps, méthode, décisions possibles)
5. **Contrôle** (revues, checkpoints)
6. **Conséquence** (retours, valorisation, apprentissages)

# **PARTIE 2 — MONTÉE EN COMPÉTENCES (1h30)**

---

# **5. Construire un plan de montée en compétences en IA**

## **5.1. Objectif**

Faire progresser chaque membre pour :

* augmenter la qualité,
* réduire les dépendances,
* améliorer la vélocité,
* augmenter la résilience de l’équipe.

## **5.2. Les 4 dimensions de compétences IA**

1. **Compétences techniques** : Python, ML, MLOps, Cloud, SQL, Pipelines, sécurité.
2. **Compétences méthodologiques** : CI/CD, Scrum, gestion du risque, documentation.
3. **Compétences comportementales** : communication, autonomie, priorisation.
4. **Compétences organisationnelles** : gestion du temps, reporting, coordination.

---

# **6. Méthode complète pour construire un plan de progression**

## **Étape 1 — Diagnostic initial**

Évaluer selon un référentiel à 4 niveaux :

* Novice
* Intermédiaire
* Confirmé
* Expert

## **Étape 2 — Identifier les “gaps”**

Exemple :

* DS excellent en modélisation mais faible en MLOps
* DE bon en SQL mais faible en gouvernance data
* PO fort en métier mais faible en compréhension IA

## **Étape 3 — Définir des objectifs SMART**

Exemples :

* “Être capable de déployer un modèle via CI/CD GitLab avant 6 semaines.”
* “Documenter tous les modèles avec un template standard avant fin du sprint.”

## **Étape 4 — Définir le plan d’actions**

* mentorat
* pair programming
* mini-projets internes
* ateliers techniques
* revues de code / MLOps
* participation à des comités d’architecture

## **Étape 5 — Suivi et réajustement**

Bimensuel ou à chaque sprint.

# **PARTIE 3 — ENTRETIENS & COMMUNICATION CORRECTIVE (1h)**

---

# **7. Les 4 types d’entretiens du chef de projet IA**

### **1. Entretien de suivi de mission**

S’assurer que la charge, les risques, les dépendances sont sous contrôle.

### **2. Entretien de motivation**

Explorer le sens du projet, clarifier les besoins, renforcer l’engagement.

### **3. Entretien de feedback**

Basé sur faits / impact / solution / plan d’actions.

### **4. Entretien correctif**

Objectif : corriger un écart, sans dégrader la relation.
Utilisation de la méthode DESC (Décrire – Exprimer – Spécifier – Conclure).

# **PARTIE 4 — PRISE DE DÉCISION EN CONTEXTE INCERTAIN (3h30)**

---

# **9. Pourquoi décider est difficile dans les projets IA ?**

* Les données sont instables.
* Les métriques évoluent après déploiement.
* Les nouveaux besoins métiers émergent.
* Le RGPD peut invalider une solution.
* Les performances modèles varient selon les distributions.
* Les coûts cloud sont imprévisibles.

Décider nécessite un cadre robuste.

---

# **10. Les outils de prise de décision**

## **10.1. Analyse SWOT appliquée à l’IA**

Exemple pour un modèle de recommandation :

* Fortes : précision élevée
* Faiblesses : faible explicabilité
* Opportunités : conversion commerciale
* Menaces : risques RGPD / biais algorithmique

---

## **10.2. Matrice Eisenhower**

Décision prioritaire selon :

* urgent / important
* non urgent / important
* urgent / non important
* non urgent / non important

Application en IA :

* Urgent/Important → Correction bug MLOps bloquant
* Important/Non urgent → Refonte pipeline data
* Urgent/Non Important → Démo commerciale
* Non urgent/Non important → Optimisation non prioritaire

---

## **10.3. Les 6 chapeaux de Bono (outil essentiel du RNCP)**

* Chapeau blanc : faits, métriques
* Rouge : émotions, perceptions
* Noir : risques
* Jaune : bénéfices
* Vert : créativité
* Bleu : organisation, décisions

Très utile quand l’équipe hésite entre plusieurs modèles IA.

---

## **10.4. Méthode PACE (IA)**

Pour choisir un modèle, un pipeline ou une architecture :

* **Performance** : précision, rappel, stabilité
* **Acceptabilité** : métier, utilisateurs, DPO
* **Coût** : cloud, maintenance
* **Effort** : développement, dette technique

---

# **11. Impliquer l’équipe dans le processus décisionnel**

## **11.1. Différence : consensus vs consentement**

* Consensus : tout le monde est d’accord
* Consentement : personne n’est contre (plus réaliste)

## **11.2. Les cercles de décision IA**

* Information
* Consultation
* Co-décision
* Décision

---

# **12. Anticiper les risques humains et organisationnels**

Les principaux risques dans un projet IA :

### **Risques humains**

* surcharge
* conflit entre experts
* démotivation
* non-communication

### **Risques techniques**

* dette data
* dette MLOps
* biais du modèle
* explicabilité insuffisante
* absence de pipeline reproductible

### **Risques organisationnels**

* dépendance à un expert clé
* documentation absente
* absence de métriques claires
* gouvernance data insuffisante

Le chef de projet doit disposer d’un **plan de mitigation**.
